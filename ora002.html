Database Performance Improvements

Reducing the number of database SQL query calls

In this post, we will look at reducing the amount of queries in the database in the first place, reducing the number of calls and decreasing the volume of data coming and going (Network load).

In the database call and performance above, call is called through the application for convenience, but the type of call is technically different.

• Parse Call: Call to request SQL parsing • Execute call: Call to execute SQL calls • Fetch call: The result of a SELECT statement that requests data transfer

• User call: Call through Application (external) calls • Recursive call: Call that occurs inside the DBMS (data dictionary lookup for SQL parsing and optimization, user-defined functions, performing SQL within the procedure)

You just have to note that call is not divided by the same classification criteria. (the kind of call varies)

This post is a light-hearted point of view of performance improvement, and we'll look at some of the ways to improve user call.

1. Using one SQL

It's easy to learn how to improve your SQL tuning so that you can load the same data into 1 SQL rather than hitting a loop in a program 100,000 times, one million times, or more SQL queries.

Or, the SQL query is called one time, but the query that fetches the same table several times internally requires tuning.

For example, suppose you want to (1) view (SELECT) which table (ex. The number of records viewed is 100,000) (2) and insert it into another table by filtering from the records you viewed.

Then it would have been 100,000 times, and the fetch call was 100k times for the lookup of 100,000 cases, and the parse call and execute call were less than 100,000 or (more than the number of insert inquiries).

This would result in a total of more than 300,000 calls to be addressed in a single SQL call (parse call, execute call).

For example, let's say you need five calls to do one job. If you change the number of calls that require 5 calls to one call at a time, there is a five-fold performance improvement when you simply calculate the operation 24 hours a day.

2. Using Array Processing

This is the concept of batch collecting, rather than insert/update/delete in the code, when working with large amounts.

Note that a series of processes must be carried out in all array units.

What I'm saying is that fetch is an array, and the next step is to handle the insert and vice versa. (All tasks should be placed to maximize the effect)

(Check the interface of the development language!)

3. Minimize Fetch Call

A DBMS cannot send a large volume of records to a client at once.

Therefore, it is possible to reduce the fetch call in this way to transfer a certain amount by dividing.

In Oracle, you can adjust a certain amount of units through the ArraySize or fetchsize settings.

For example, if ArraySize is set to 100 and you read 30,000 records, fetch call will occur at 301.

OLTP is a good practice when you have a lot of query results (records), but you only want to fetch some of them.

(This is referred to as partial-scope processing, because the user does not read all tens of thousands of cases)

Note that the DBMS does not have any performance improvements when it is programmed in such a way that partial-scope processing requires that all values be fetched and processed by the developer.

Ultimately, the point is that you need to adjust the arraysize accordingly to the amount of data that needs to be transmitted over the network.

-If you need to get all your data, it is better to arraysize. Although the amount is intact, the number of fetch call decreases.

-If you need only a small fraction of the data in front of it, it is better to arraysize.

Increasing the arraysize reduces network load and also reduces the number of blocks that the server process should read.

The reason is that there are 10 records in one block, and two blocks, that is, 20 records, when you need to read all 20 records, if ArraySize is 3, block I/O occurs eight times, fetch call takes place seven times.

If ArraySize is 5, the block I/O occurs four times and fetch call also happens four times.

However, ArraySize is not a good thing to raise. ArraySize and call numbers are inversely proportional (?) , so test and use it properly.

4. Use of page processing

Improving performance with partial-scope processing is only possible in the development environment of expressions that fetch only when the user explicitly requests without closing the cursor.

In a Web application environment that does not persist connections with the database, it is not possible to keep the cursor open and handle the processing of an expression that performs an individual SQL statement whenever the user requests the next page.

5. Adult a distributed query, filtering only minimal data when remote joins

If you have created a distributed database system, you may need to send and join tables in the remote.

If there are 500,000 records in the table that should be joined, the network load will be sent and received for 500,000.

This will also result in poor query performance, so you should use a query to split the required parts.

6. Use user-defined functions, procedure only when absolutely necessary

User-defined functions and procedure that are performed inside a DBMS are not fully compiled as Native code, like intrinsic functions, and run on separate engines such as virtual machines.

This causes the context switching to occur each time it runs, which degrades performance compared to intrinsic functions.

Additionally, a user-defined function with another built-in query statement falls further.

Instead of recommending the end of the writing, you should find a suitable time.

When appropriate, it is time to call a small amount and perform a large amount of SQL from the inside.
